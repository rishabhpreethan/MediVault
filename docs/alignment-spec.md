# MediVault — Alignment Specification

Used by the **Reviewer Agent** to evaluate every PR against the SRS, architecture decisions, and project principles.

---

## Purpose

This spec is the checklist against which all generated code is reviewed. A PR cannot receive reviewer sign-off unless every applicable item in this spec is satisfied. If an item doesn't apply to the PR (e.g., a frontend-only PR won't have DB changes), mark it N/A.

---

## 1. Security & Privacy Checklist

### 1.1 PHI Handling
- [ ] `full_name` and `date_of_birth` in `family_members` are encrypted before DB write using the application-layer `ENCRYPTION_KEY` (AES-256)
- [ ] No PHI (patient names, medical values, document content) appears in any log statement
- [ ] No PHI appears in exception messages that could surface in API error responses
- [ ] All MinIO storage paths use the `{user_id}/{member_id}/` prefix structure — no shared paths
- [ ] Raw IP addresses are never stored; `passport_access_log` stores `SHA-256(ip)` only

### 1.2 Authentication & Authorization
- [ ] Every private API endpoint validates the Auth0 JWT via the `get_current_user` dependency
- [ ] Every data-access query scopes results to `user_id` from the JWT — never accepts `user_id` as a query param from the client
- [ ] Family member access is verified: user can only access `member_id`s linked to their `user_id`
- [ ] Public passport endpoint `/passport/public/{uuid}` requires **no** auth — verify it has no auth dependency
- [ ] Passport UUIDs are generated by PostgreSQL `gen_random_uuid()` — never sequential or predictable

### 1.3 Data Transmission
- [ ] No PHI is transmitted to any external API during document processing (CON-006 from SRS)
- [ ] If any external API is introduced, it is flagged in the PR and discussed against CON-006
- [ ] All API responses use HTTPS (enforced at infrastructure level in Docker Compose + Nginx)

### 1.4 Input Validation
- [ ] File uploads: validate MIME type server-side (not just extension), check file size ≤ 20MB
- [ ] Virus scan (ClamAV) runs before any file processing or storage
- [ ] Embedded text layer check runs before enqueuing extraction job
- [ ] All Pydantic models have explicit type annotations and field validators — no `Any` types for PHI fields
- [ ] SQL queries use ORM (SQLAlchemy) — no raw string SQL that could be injection vectors

### 1.5 Rate Limiting
- [ ] Auth endpoints: rate limited (10 req/min per IP)
- [ ] Upload endpoint: rate limited (20 req/min per user)
- [ ] Public passport endpoint: rate limited (60 req/min per IP)

## 2. Architecture Alignment Checklist

### 2.1 Stack Conformance
- [ ] Backend uses Python + FastAPI + Pydantic v2 — no other frameworks introduced
- [ ] PDF extraction uses pdfminer.six as primary, pypdf as fallback — no other extraction library introduced without a 2-Way Door decision
- [ ] NLP uses spaCy + Med7 — no external NLP API calls — satisfies CON-006
- [ ] Auth uses Auth0 JWT (RS256) — no custom JWT signing
- [ ] Object storage uses MinIO via S3-compatible API (`boto3` or `aioboto3`) — no filesystem storage of uploaded files
- [ ] Task queue uses Celery with Redis broker — no other queue system
- [ ] Database ORM uses SQLAlchemy (async) — Alembic for migrations

### 2.2 Data Model
- [ ] All primary keys are UUIDs (`gen_random_uuid()`) — no auto-increment integers for user-facing entities
- [ ] All medical entity tables include `member_id` FK (not `user_id` directly) — data scoped to family member
- [ ] All entity tables include `document_id` FK (nullable) — links extracted data to source document
- [ ] All extracted entities have `confidence_score` field (HIGH/MEDIUM/LOW)
- [ ] All extracted entities have `is_manual_entry` boolean
- [ ] Alembic migration file is included for any DB schema change
- [ ] No migration drops or renames a column without a backward-compatible transition step

### 2.3 API Design
- [ ] All endpoints versioned under `/api/v1`
- [ ] Response schemas use Pydantic models — no raw dicts returned
- [ ] Errors return structured JSON: `{ error: string, message: string }`
- [ ] Upload endpoint returns 202 Accepted (not 200) — processing is async
- [ ] Deletion endpoints return 204 No Content
- [ ] List endpoints are paginated (`page`, `limit` query params; default limit ≤ 50)

### 2.4 Async Processing
- [ ] Document processing is always async — never blocks the upload HTTP response
- [ ] Celery tasks handle retries (max 3 attempts) with exponential backoff
- [ ] Task failures update `documents.processing_status` to `FAILED` after max retries
- [ ] No Celery task directly returns PHI data — tasks communicate via DB state

## 3. SRS Functional Requirements Checklist

For each feature area, verify the critical FRs are met.

### 3.1 Document Upload (SRS §3.2)
- [ ] Accepts only PDFs ≤ 20MB (FR-DOC-001, FR-DOC-002)
- [ ] Detects and rejects scanned PDFs with user-friendly message (FR-DOC-003)
- [ ] Supports multi-file upload up to 10 files (FR-DOC-004)
- [ ] Shows real-time upload progress (FR-DOC-005)
- [ ] User assigns document type and date (FR-DOC-006, FR-DOC-007)
- [ ] Original PDF stored and downloadable (FR-DOC-008)
- [ ] Document library with status badges (FR-DOC-009)
- [ ] Virus scan on all uploads (FR-DOC-012)

### 3.2 PDF Extraction (SRS §3.3)
- [ ] Text extraction runs server-side, no external API (FR-EXT-001, FR-EXT-010)
- [ ] Embedded text detection before processing (FR-EXT-002, FR-EXT-003)
- [ ] Async with status updates (FR-EXT-004, FR-EXT-005)
- [ ] Retry up to 3 times (FR-EXT-006)
- [ ] No PHI leaves infrastructure during extraction (FR-EXT-010)

### 3.3 NLP Extraction (SRS §3.4)
- [ ] Medications: name, dosage, frequency, duration (FR-NLP-001)
- [ ] Lab results: test name, value, unit, reference range (FR-NLP-002)
- [ ] Diagnoses: condition name and date (FR-NLP-003)
- [ ] Lab flags: H/L indicator for out-of-range values (FR-NLP-007)
- [ ] Deduplication of chronic conditions (FR-NLP-008)
- [ ] Confidence scores assigned (FR-NLP-010)
- [ ] Low-confidence fields flagged for review (FR-NLP-011)
- [ ] Manual edit/delete of any extracted field (FR-NLP-012)

### 3.4 Health Profile (SRS §3.5)
- [ ] Summary card (FR-PROF-001)
- [ ] Consolidated medications list (FR-PROF-002)
- [ ] Consolidated conditions list (FR-PROF-003)
- [ ] Allergies list with severity (FR-PROF-004)
- [ ] Source document attribution per data point (FR-PROF-009)
- [ ] Distinction between auto-extracted and manually entered (FR-PROF-008)

### 3.5 Health Passport (SRS §3.8)
- [ ] Shareable URL, no login required to view (FR-PASS-001)
- [ ] QR code generated (FR-PASS-002)
- [ ] Patient-controlled revocation (FR-PASS-003)
- [ ] Clearly labelled as patient-reported (FR-PASS-006)
- [ ] Access logged (FR-PASS-007)

## 4. Non-Functional Requirements Checklist

### 4.1 Performance (SRS §4.1)
- [ ] Upload API responds within 1 second of file submission (NFR-PERF-002)
- [ ] Extraction pipeline completes within 15 seconds for 95% of standard PDFs (NFR-PERF-003)
- [ ] Profile/timeline API responds within 1.5 seconds (NFR-PERF-004)

### 4.2 Reliability
- [ ] Document loss prevention: failed extractions are retryable, original PDF never deleted on failure (NFR-REL-002)
- [ ] No orphaned records: deleting a document cascades to all extracted entities

### 4.3 Usability
- [ ] All interactive elements have minimum 44×44px touch target (NFR-USE-004)
- [ ] Mobile-first: all screens designed for 375px viewport up (NFR-USE-001)
